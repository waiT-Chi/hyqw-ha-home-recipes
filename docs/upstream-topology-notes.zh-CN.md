# 上游拓扑思路：本地 MQTT 接管但保留原厂链路

这篇文档整理的是上游/社区方案里的网络拓扑思路。它不复制原图，也不发布任何真实域名、IP、账号、密码、topic 或 payload。

## 核心目标

地产 485 网关原本会连接开发商云端。上游方案的核心是：

```text
485 网关
  -> 被本地 DNS/hosts 规则引导到本地 MQTT
  -> 本地 MQTT bridge 再桥接到开发商 MQTT
  -> Home Assistant 从本地 MQTT 侧接入和生成实体
```

这样可以同时获得几个效果：

- Home Assistant 本地控制响应更快；
- 原厂 App 和中控屏仍可继续使用；
- 485 网关状态可以进入 HA；
- 后续可以把 Apple Home、米家、语音音箱等用户入口接到 HA 上。

## 上游拓扑里的关键角色

### 1. 路由 / OpenWrt 层

上游方案通常会让 OpenWrt 接管 DHCP 和 DNS 解析能力，而不是只把它当普通旁路设备。

它主要负责：

- 给家庭网络分配地址；
- 把 485 网关访问的开发商 MQTT 域名重定向到本地 MQTT；
- 维持普通设备上网和原厂 App 可用；
- 可选地承载 OpenClash、ZeroTier、NAS 等旁路能力。

### 2. 本地 MQTT 层

本地 MQTT 是接管链路的核心。

它通常做两件事：

- 接收 485 网关被重定向过来的 MQTT 连接；
- 建立 bridge，把必要消息继续透传到开发商 MQTT。

这层的关键是“接管但不断链”：不要让网关只能连本地，也不要让原厂 App 和中控屏失效。

### 3. Home Assistant / bridge 层

HYQW adapter 或前湾 bridge 连接本地 MQTT，在 Home Assistant 中生成灯光、空调、地暖、窗帘、新风等实体。

随后 HA 再负责：

- 自动化；
- dashboard；
- Apple Home / HomeKit Bridge；
- 米家生态整合；
- 语音入口；
- 通知和状态验证。

## 与本仓库参考部署的关系

上游拓扑更强调网络侧接管：OpenWrt、DNS/hosts、MQTT bridge、PVE/工控机等。

本仓库的参考部署更强调家庭运维落地：

- 使用 Mac mini + Docker 承载 Home Assistant 和周边服务；
- 把 HYQW adapter 作为地产 485 接入层；
- 再把米家、美的、Apple Home / Siri、自动化和恢复检查纳入同一个控制平面；
- 记录控制边界、迁移策略、可靠性和隐私去敏。

两者并不冲突：

```text
上游拓扑解决：485 网关流量如何本地化接入 HA
本仓库补充：接入 HA 之后，如何把全屋多生态真正用起来
```

## 复用时的注意事项

不要直接照抄别人的拓扑。开始前至少确认：

- 你的主路由、AC、AP、旁路由分别负责什么；
- DHCP 只由一个明确的设备负责；
- DNS 重定向只影响 485 网关需要接管的域名；
- 本地 MQTT bridge 不会形成消息环路；
- 原厂 App 和中控屏仍能正常控制；
- HA 宕机时，基础照明、窗帘、空调等本地生活能力不会完全瘫痪。

如果不确定，先只做只读状态接入和小范围测试，不要一开始就接管全屋控制。

## 不应该公开的内容

公开文档里不要放：

- 真实开发商 MQTT 域名和端口；
- MQTT 用户名、密码和 client id；
- 485 网关 SN；
- 完整 hosts / DNS 劫持规则；
- replay payload；
- 家庭网络 IP 拓扑；
- 能直接复现控制链路的完整配置包。

建议公开的是思路、边界、检查表和去敏模板，而不是完整可复制的私有网络配置。
